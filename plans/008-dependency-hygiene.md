## Why this matters

Two dependency cleanups with immediate benefit:

1. **nodemailer ^8.0.7** has 3 known CVEs: CRLF injection in List-\* headers (HIGH severity, GHSA-268h-hp4c-crq3), `jsonTransport` bypasses `disableFileAccess`/`disableUrlAccess` (GHSA-wqvq-jvpq-h66f), and improper TLS cert validation in OAuth2 token fetch (GHSA-r7g4-qg5f-qqm2). The fix is a semver-patch bump to `^8.0.11`.

2. **node-fetch ^3.3.1** is an unnecessary third-party dependency. Node 24 (required by this project) has a stable global `fetch` API. The frontend already uses `global.fetch` directly. Removing `node-fetch` eliminates a dependency with known CRLF injection vulnerability (GHSA-hmw2-7cc7-3qxx in `form-data`, a transitive dep of `node-fetch`).

## STOP conditions

Stop and report back if:

- Removing `node-fetch` causes any test to fail — some test may import `node-fetch` directly for mocking. If so, update the test to mock the global `fetch` instead (pattern: `globalThis.fetch = mockFetch`).
- The typecheck step fails with "Cannot find name 'fetch'" — Node 24's type definitions include `fetch` globally; if tsconfig somehow excludes it, add `"DOM"` to `lib` in the server tsconfig or cast as needed.

## Maintenance notes

- Node 24's built-in `fetch` is stable and spec-compliant. The one behavioral difference from `node-fetch`: `node-fetch` v3 is ESM-only and has custom agent support. The built-in `fetch` uses the global `http.Agent`. If any price provider code was relying on custom agent configuration (unlikely), it will break. Verify by running the price tests.
- If a future Node version deprecates the global `fetch`, the import can be restored — but that's unlikely given `fetch` is a web standard.
